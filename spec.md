# BRIEF BASE DEL PROYECTO
# Plataforma de formación personalizada para aprender Figma, UI/UX y Product Design desde cero hasta nivel profesional

## 1. Idea general

Crear una plataforma web de aprendizaje personalizada donde el usuario pueda seguir un bootcamp completo de Figma, UI/UX y Product Design desde nivel cero.

La plataforma debe estar pensada para una persona que:
- Parte casi desde cero en Figma.
- Quiere aprender de forma seria.
- Quiere llegar a defenderse profesionalmente.
- Quiere trabajar como diseñador UI/UX.
- Quiere poder hacer freelancing en el futuro.
- Quiere dominar Figma, diseño visual, UX, sistemas de diseño, prototipado, handoff y producto digital.
- Prefiere aprender con mini teoría + reto práctico.
- Odia que los cursos no se parezcan a la realidad profesional.
- Es perfeccionista, visual, procrastinador y algo competitivo.
- Prefiere dominar cada fase antes de avanzar.

La plataforma no debe sentirse como un curso genérico de tutoriales, sino como un entorno de entrenamiento profesional guiado.

---

## 2. Filosofía del curso

El objetivo no es enseñar “botones de Figma”.

El objetivo es formar a una persona capaz de pensar como diseñador de producto digital.

La plataforma debe enseñar:

- Criterio visual.
- Uso profesional de Figma.
- UI Design.
- UX realista.
- Sistemas de diseño.
- Componentes.
- Variables.
- Auto Layout.
- Responsive design.
- Prototipado.
- Handoff a desarrollo.
- Dev Mode.
- Organización profesional.
- Trabajo con restricciones reales.
- Mentalidad de producto.
- Preparación para portfolio.
- Preparación para freelancing.

El perfil objetivo final es:

“Diseñador/a de producto digital con mentalidad frontend.”

---

## 3. Estilo de aprendizaje

El curso debe seguir esta estructura en cada lección:

1. Mini teoría
   - Breve.
   - Clara.
   - Directamente relacionada con el reto.
   - Sin teoría innecesaria.
   - Sin “UX humo”.

2. Reto práctico
   - Realista.
   - Parecido a encargos profesionales.
   - Con restricciones claras.
   - Con objetivos medibles.

3. Entrega
   - El usuario sube capturas, enlace de Figma o descripción del resultado.

4. Evaluación tipo mentor
   - Feedback profesional.
   - No destructivo, pero exigente.
   - Señalar errores concretos.
   - Proponer mejoras.
   - No avanzar hasta que la fase esté dominada.

5. Reintento
   - El usuario mejora el diseño.
   - La plataforma guarda versiones.
   - Se puede comparar progreso.

---

## 4. Tono de la plataforma

El tono debe ser:

- Profesional.
- Directo.
- Motivador.
- Realista.
- Nada infantil.
- Nada académico tradicional.
- Nada de “curso barato”.
- Más parecido a tener un mentor senior que revisa tu trabajo.

Ejemplo de tono:

“No estás aprendiendo a decorar pantallas. Estás aprendiendo a construir productos digitales claros, usables y escalables.”

---

## 5. Problema principal del usuario

El usuario tiene potencial, pero su mayor riesgo es:

Perfeccionismo + procrastinación.

La plataforma debe combatir esto con:

- Restricciones de tiempo.
- Retos pequeños.
- Checkpoints claros.
- Progreso visible.
- Evaluaciones por rúbrica.
- Bloqueo de avance hasta dominar conceptos clave.
- Sistema de versiones.
- Objetivos concretos.
- Retos competitivos opcionales.
- Métricas de mejora.

## 8. Sistema de bloqueo de avance

La plataforma debe permitir avanzar solo si:

- El usuario completa el reto.
- El reto ha sido evaluado.
- La nota mínima se supera.
- Las mejoras obligatorias están resueltas.

Notas sugeridas:

- 0-4: Rehacer.
- 5-6: Mejorar.
- 7: Aprobado justo.
- 8: Buen nivel.
- 9: Muy buen nivel.
- 10: Nivel excelente para la fase.

Para avanzar de fase:
- Nota media mínima: 8.
- Ningún criterio crítico por debajo de 7.
- Entrega final aprobada.

---

## 9. Sistema contra procrastinación

La plataforma debe incluir:

- Retos con tiempo recomendado.
- Modo “entrega imperfecta”.
- Recordatorio: “Primero termina, luego perfecciona”.
- Historial de versiones.
- Comparación antes/después.
- Objetivos diarios.
- Racha semanal.
- Microretos de 15-30 minutos.
- Recompensas por entregar, no solo por perfeccionar.
- Avisos si el usuario pasa demasiado tiempo sin entregar.

---

## 10. Sistema competitivo opcional

Como el usuario es algo competitivo, incluir:

- Puntuación por reto.
- Nivel de diseñador.
- Insignias.
- Ranking personal contra versiones anteriores.
- Desafíos cronometrados.
- Objetivo de superar nota anterior.
- Medallas por:
  - Mejor spacing.
  - Mejor jerarquía.
  - Mejor uso de Auto Layout.
  - Mejor componente.
  - Mejor prototipo.
  - Mejor handoff.

No es necesario ranking público al principio.

## 21. Deployment

SkillDrop se despliega como **multi-container Docker Compose** con un único servicio
público (`web`, nginx) que sirve el frontend y hace de reverse-proxy de `/api` hacia
el servicio interno `api` (Express). El servicio `api` no se publica al host.

```yaml
deployment:
  mode: docker-compose
  public_service: web
  internal_port: 80
  healthcheck_path: /
```

- **mode:** docker-compose
- **public_service:** web
- **internal_port:** 80
- **healthcheck_path:** /

Notas:
